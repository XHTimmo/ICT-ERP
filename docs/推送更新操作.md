# 推送更新操作说明

本文档用于规范本项目发布新版本并触发自动更新的操作流程。

## 1. 发布前准备

1. 确认本地代码已完成并通过构建：
   - `npm run build`
2. 更新更新日志文件：
   - 根目录 `RELEASE_NOTES.md`（GitHub Release 描述来源）
   - `docs/erp_vX.Y.Z.md`（版本技术文档）
   - 将本次更新内容同步到软件发布说明（GitHub Release 描述）并保持与 `RELEASE_NOTES.md` 一致
3. 确认版本号策略：
   - 标签格式必须为 `v*`（例如 `v1.3.5`）
   - GitHub Actions 会按标签自动写入 `package.json` 版本号，无需手改并提交

## 2. 推送发布

在仓库根目录执行以下命令：

```bash
git add .
git commit -m "release: v1.3.5"
git push origin main
git tag v1.3.5
git push origin v1.3.5
```

说明：
- 推送 `v*` 标签后，会触发 `.github/workflows/release.yml`。
- 工作流会先创建 GitHub Release，再在 macOS/Windows 构建并发布安装包与更新元数据。

## 3. 发布后检查

1. 在 GitHub Actions 查看 `Release` 工作流是否成功。
2. 在 GitHub Releases 检查是否包含以下产物（按平台）：
   - macOS: `.dmg`、`.zip`、`latest-mac.yml`
   - Windows: `.exe`、`latest.yml`
3. 在 GitHub Releases 检查发布说明是否已同步本次更新内容，并与 `RELEASE_NOTES.md` 一致。
4. 启动旧版本客户端，检查：
   - 可检测到新版本
   - 可下载更新并提示重启安装

## 4. 失败排查清单

1. 自动更新 404：
   - 检查仓库是否为 Public
   - 检查 `package.json` 中 `repository` 与 `build.publish` 的 `owner/repo` 是否一致
2. 未触发发布：
   - 检查标签是否以 `v` 开头（如 `v1.3.5`）
3. Release 无内容：
   - 检查 `RELEASE_NOTES.md` 是否存在且已提交
4. 客户端不提示更新：
   - 确认新版本号高于当前版本
   - 确认目标机器可访问 GitHub Releases

## 5. 软件发布说明模板

每次发布时，将以下模板复制到 GitHub Release 描述，并按实际内容替换占位符，确保与 `RELEASE_NOTES.md` 保持一致。

```md
## ERP vX.Y.Z

### ✨ 新增
- 新增：<功能点 1>
- 新增：<功能点 2>

### 🛠 优化
- 优化：<优化点 1>
- 优化：<优化点 2>

### 🐞 修复
- 修复：<问题 1>
- 修复：<问题 2>

### 📦 安装包
- macOS: `.dmg`、`.zip`
- Windows: `.exe`

### 🔄 自动更新
- 支持从旧版本检测并升级到 `vX.Y.Z`
```
